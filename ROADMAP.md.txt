```bash
# // - Serde: #[serde(rename_all = "camelCase")]
```

- **DO NOT** create separate binaries unless the phase explicitly says to — library crates only
- **DO NOT** use `unwrap()` or `expect()` in library code — propagate errors

- **DO** run `cargo check` after every file you create or modify
- **DO** run `cargo clippy` before marking a phase complete


### Protocol Specification

**Request frame** (client → server):
```json
{"type": "req", "id": "<uuid>", "method": "<string>", "params": {...}}
```

**Response frame** (server → client):
```json
{"type": "res", "id": "<uuid>", "ok": true, "payload": {...}}
// or on error:
{"type": "res", "id": "<uuid>", "ok": false, "error": {"code": "<string>", "message": "<string>"}}
```

**Event frame** (server → client, unchanged but add sequence):
```json
{"type": "event", "event": "<string>", "payload": {...}, "seq": <number>}
```

### Tasks (Ordered)

1. **Read** `ws.rs` completely — understand current message handling
2. **Create** `crates/rune-gateway/src/ws_rpc.rs`:
   - Define `WsRequest`, `WsResponse`, `WsEvent` structs with serde
   - RPC method dispatcher: maps method strings to existing service/handler calls
   - Sequence number counter (atomic u64)
   - Error types for RPC failures
3. **Modify** `crates/rune-gateway/src/ws.rs`:
   - Parse incoming frames: if `type == "req"` → route through RPC dispatcher
   - Keep existing event broadcast, add `seq` field
   - Add stateVersion tracking for connection health

## Phase 4 — LaneQueue Concurrency Model (Backend)

**Priority:** High — prevents state corruption
**Dependencies:** None
**Estimated scope:** 1 new file, 1 modified file

### Reference Files to Study

```
# From OpenClaw — study the LaneQueue specification:
openclaw/openclaw: search for "LaneQueue", "lane", "concurrency", "serial"
# Extract: lane types, cap values, queuing behavior, "Default Serial Explicit Parallel"

# From ZeroClaw — study their concurrency model:
zeroclaw: search for concurrency, queue, semaphore patterns
```

### Step 0: Read Existing Code

```bash
cat crates/rune-runtime/src/executor.rs     # Current execution model — this is what you're replacing
# Understand: How are turns currently executed? What's the entry point?
# Find: The function that receives a message and runs the LLM turn
```

### Design

```
Lane enum:
  Main      — cap 4  (user-initiated turns)
  Subagent  — cap 8  (spawned sub-agent turns)
  Cron      — independent (heartbeat/scheduled, no cap sharing)
  Nested    — recursive (tool calls within a turn, inherits parent lane)

Implementation:
  Per-lane: tokio::sync::Semaphore(cap) + VecDeque<PendingTask>
  FIFO ordering: tasks dequeue in submission order
  Task submission returns a JoinHandle-like future
  Lane routing: based on session kind (main session → Main, sub-agent session → Subagent, etc.)
```

### Tasks (Ordered)

1. **Read** `executor.rs` — understand the current turn execution flow
2. **Create** `crates/rune-runtime/src/lane_queue.rs`:
   - `Lane` enum with `Main`, `Subagent`, `Cron`, `Nested` variants
   - `LaneQueue` struct with per-lane `Semaphore` + `VecDeque`
   - `submit(lane, task) -> impl Future<Output = Result<T>>` method
   - Lane routing function: `fn route_lane(session: &Session) -> Lane`
3. **Modify** `crates/rune-runtime/src/executor.rs`:
   - Replace direct execution with `lane_queue.submit(lane, turn_task)`
   - Keep the existing turn execution logic inside the task closure

### Verification

```bash

cargo test -p rune-runtime
# Write a unit test that submits 6 Main tasks and verifies only 4 run concurrently
```

## Phase 10 — Hybrid Memory Search (Backend)

**Priority:** High — major capability gap
**Dependencies:** None
**Estimated scope:** 1 migration, 1 new file, 2 modified files

### Reference Files to Study

```

opencrust: search for "pgvector", "vector", "embedding", "memory"
# IronClaw also uses pgvector — study their Diesel model for vector type

# From ZeroClaw — study hybrid search algorithm:
zeroclaw: search for "memory", "vector", "FTS5", "hybrid"
# They use: 70% vector cosine similarity + 30% FTS5 BM25 keyword
# Stored in SQLite with sqlite-vec extension

# Rune uses PostgreSQL — we adapt to pgvector + tsvector instead
```

### Step 0: Read Existing Code

```bash
cat crates/rune-tools/src/memory_tool.rs    # Current keyword-only search — you're upgrading this
cat crates/rune-store/src/                  # Database layer — check for migration patterns
```

### Tasks (Ordered)

1. **Create migration:**
   ```sql
   CREATE EXTENSION IF NOT EXISTS vector;
   
   CREATE TABLE memory_embeddings (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     file_path TEXT NOT NULL,
     chunk_text TEXT NOT NULL,
     embedding vector(1536),  -- OpenAI text-embedding-ada-002 dimension
     updated_at TIMESTAMPTZ DEFAULT now()
   );
   
   CREATE INDEX ON memory_embeddings USING ivfflat (embedding vector_cosine_ops);
   CREATE INDEX ON memory_embeddings USING gin (to_tsvector('english', chunk_text));
   ```

2. **Create** `crates/rune-tools/src/memory_index.rs`:
   - Embedding provider abstraction (OpenAI text-embedding-ada-002 or configurable)
   - File chunking: split by heading/paragraph, ~512 token chunks
   - Batch embed + upsert to memory_embeddings table
   - Background re-index on workspace file watcher events (use `notify` crate)

3. **Modify** `crates/rune-tools/src/memory_tool.rs`:
   - Replace word-hit scoring with hybrid PostgreSQL query:
     ```sql
     SELECT *, 
       (0.3 * ts_rank(to_tsvector('english', chunk_text), plainto_tsquery($1))) +
       (0.7 * (1 - (embedding <=> $2::vector))) AS score
     FROM memory_embeddings
     ORDER BY score DESC
     LIMIT $3
     ```
   - Reciprocal Rank Fusion (RRF) to merge keyword + vector scores
   - Lazy re-indexing: on file change → re-chunk → re-embed → upsert

4. **Modify** `crates/rune-store/` — add MemoryEmbeddingRepo trait + model

5. **Add dependency:** `pgvector` crate (for Diesel vector type support)
